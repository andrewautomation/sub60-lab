## Known limitations (by design, not oversights)

3. **Duathlon and Aquathlon get no Race Prediction or ROI-by-Discipline.** The underlying prediction engine (`lib/race/*`) only has leg-target models for triathlon formats. Building a second multisport pacing/transition model for two more sports was explicitly out of scope ("no new concepts"). Both outputs return a clear `supported: false` with an explanation rather than a number computed from the wrong model â€” this is honest degradation, not a bug.
4. **Goal Confidence's "pace-to-deadline" component uses a flat, stated assumption** (a 84-day/~12-week minimum training block), not a rate derived from the athlete's own historical improvement. Documented explicitly in `lib/performance-engine/goalConfidence.ts` and surfaced in the engine's own `assumptions` array â€” this is v1 decision support, not a fitted model, per the original request's own framing ("useful, transparent decision support," not "perfect prediction").
5. **"Gap to Goal" on the Swim/Bike/Run module pages uses this app's generic Sub-60 Sprint defaults**, not the athlete's actual saved goal â€” only the Performance Engine's Goal Gap output uses the real goal. Unifying these is real future work (see `ROADMAP.md`'s Goals epic notes), not done here to avoid touching already-shipped, already-tested module pages mid-RC.
6. **No equipment or baseline data collection anywhere in the app.** Dropped from onboarding's scope in an earlier pass ("collect only the minimum required information") and never reintroduced. `services/equipment.service.ts` and `services/performanceProfile.service.ts` exist, match a real DB schema, and are entirely unused â€” safe to keep as-is until/unless this scope is revisited.
7. **No structured feedback collection for the Performance Engine's three reflection questions.** They're displayed as static text under the Performance Engine section. Persisting an athlete's answers would need a new database table â€” a genuinely new concept, explicitly out of scope for this Release Candidate pass.
8. **No server-side route protection (`middleware.ts`).** Protection is a client-side gate in `app/dashboard/layout.tsx` covering the whole `/dashboard/*` subtree â€” correct in that a signed-out user is always redirected, but a signed-out user's browser briefly renders a loading state before the redirect fires, rather than never receiving the page at all. Fixing this properly means moving off the current browser-only `@supabase/supabase-js` client onto `@supabase/ssr`'s cookie-based session handling â€” a real architecture change, explicitly out of scope ("do not redesign the architecture"). Supabase RLS prevents any actual data leakage regardless.
9. **No automated test suite.** Every verification in this project has been manual: `tsc`/`eslint`/`build` for static correctness, live Playwright-driven browser checks for auth-gating and page rendering, and (for the Performance Engine specifically) a hand-traced synthetic-data smoke test with every number independently recomputed by hand. Adding Vitest/Jest is a real dependency and config decision, flagged here rather than assumed. The Performance Engine's modules are pure functions with no Supabase/React dependency, so they're ready to test the moment a framework is chosen.
